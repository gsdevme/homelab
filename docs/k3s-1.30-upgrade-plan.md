## Confirmed environment (Phase 0, done)

- **Running k8s: `v1.26.15+k3s1`** (not 1.27 as install.sh implied) → **5 hops** to 1.30:
  1.26→1.27→1.28→1.29→1.30.
- **Topology:** 1 server `kube-master` (amd64, `control-plane,etcd,master`, **embedded etcd**) +
  3 agents `pi-alpha`/`pi-beta`/`pi-gamma` (arm64 Pis). Single control-plane → brief API downtime
  during the server's own upgrade; embedded etcd → snapshot + `--cluster-reset` restore is valid.
- **Tooling:** kubectl v1.36.2, **flux CLI v2.7.5** (too new — generated manifests want k8s ≳1.31;
  when bumping Flux, pin an intermediate: `flux install --version=v2.4.x --export`, don't just
  regenerate with the local CLI).
- **SUC pinned v0.19.2.** Its `default-controller-env` ConfigMap sets
  `SYSTEM_UPGRADE_JOB_KUBECTL_IMAGE: rancher/kubectl:v1.30.3` — a large kubectl↔apiserver skew on
  the first (1.26→1.27) hop. Drain uses the stable eviction API so it should be fine, but if drain
  misbehaves in Phase 3, override that image to a nearer version for the early hops.

## Context

The homelab cluster is running an old Kubernetes (CLAUDE.md says ~1.26; the historical
`k3sup/home-cluster/install.sh` pins `v1.27.16+k3s1` — **actual running version must be
confirmed**). The goal is to move up the Kubernetes minor versions to **1.30** (a widely
battle-tested target with comfortable component-support overlap; 1.32 deferred to a later pass).

Two realities shape this plan:

1. **This repo does not currently control the k3s version.** The k3sup commands in
   `install.sh` are commented out and purely historical. We make the version **declarative** by
   adding Rancher's **system-upgrade-controller (SUC)** as an infrastructure component and driving
   upgrades with `Plan` CRDs committed to git — matching how everything else here deploys.

2. **Several pinned components will not run on newer Kubernetes** and must be upgraded as part
   of this work:
   - `cert-manager 1.8.2` (supports k8s ≤1.24)
   - `ingress-nginx 4.2.5` / controller v1.3 (supports k8s ≤1.25)
   - `Flux v2.0.0` with `HelmRelease v2beta1` (officially supports k8s ≤1.28)

Because component support windows and k8s versions must overlap at every step, the upgrade is
**interleaved and sequential**, not a single jump.

## Guiding constraint: support-window overlap

| Stage | k8s | Flux | cert-manager | ingress-nginx |
|-------|-----|------|--------------|---------------|
| Now | ~1.27 | 2.0.0 | 1.8.2 | 4.2.5 (v1.3) |
| After component bumps (still on current k8s) | ~1.27 | 2.0.0 | **1.14.x** (1.27–1.29) | **4.11.x** / v1.11 (1.26–1.30) |
| Mid-step | 1.28 | **2.3.x/2.4.x** (1.28–1.31) | 1.14.x | 4.11.x |
| Target | **1.30** | 2.3.x/2.4.x | **1.15.x/1.16.x** (1.28–1.30) | 4.11.x |

Newer cert-manager/ingress-nginx charts run fine on the *current* k8s, so we bump them **first**
(while the cluster is stable), then step k8s, bumping Flux once k8s reaches 1.28.

## Phase 0 — Prep & safety (no cluster mutations yet)

1. **Confirm ground truth** (read-only):
   - `kubectl version` and `kubectl get nodes -o wide` — actual k8s version, node count, arch.
   - Determine **server vs agent** node counts (install.sh implies a single server behind a
     kube-vip VIP + 3 agents). Confirm; it decides SUC Plan node-selectors and control-plane
     downtime expectations.
2. **Backups:**
   - k3s etcd snapshot on the server: `k3s etcd-snapshot save` (and note snapshot location).
   - Export cert-manager state: `kubectl get clusterissuer,issuer,certificate -A -o yaml` to a
     local file (API is `cert-manager.io/v1`, stable — resources survive the chart bump, but keep

   - Note PVCs: `apps/prod/tesla` postgres uses **local-path** (node-local) storage pinned to an
     **arm64** node. Draining that node during upgrade briefly stops postgres; data persists on
     the node (in-place upgrade keeps node identity). Accept brief downtime.
3. **Hygiene fix (inert, safe):** `apps/no-deployment/glasgow-bin-collection/cronjob.yaml` uses
   removed `batch/v1beta1` → change to `batch/v1`. It's under `no-deployment/` so not live, but
   fix it so it's not a future landmine.

## Phase 1 — Add system-upgrade-controller (GitOps)

- Create `infrastructure/base/system-upgrade-controller/` containing SUC's vendored manifests
  pinned to a release tag (e.g. `v0.14.x`): the `system-upgrade` namespace, CRDs
  (`plan.upgrade.cattle.io`), RBAC, and the controller Deployment. Source: Rancher's
  `system-upgrade-controller.yaml` + `crd.yaml` for the pinned tag. Add a `kustomization.yaml`
  listing them (native-manifest style, like other components).
- Add `system-upgrade-controller` to the allow-list in
  `infrastructure/base/kustomization.yaml` (`resources:`), alongside `cert-manager`, `nginx`, etc.
- Commit; let Flux reconcile; verify the controller pod is Running before creating any Plan.

## Phase 2 — Bump cert-manager & ingress-nginx (while on current k8s)

These newer charts support the current k8s, so do them before touching k8s to isolate risk.

- **cert-manager** `infrastructure/base/cert-manager/release.yaml`: `version: "1.8.2"` →
  `"1.14.x"`. Keep `installCRDs: true` (the `crds.enabled` rename lands in 1.15 — handle when we
  go to 1.15 in Phase 3). cert-manager recommends stepwise minor upgrades; the `cert-manager.io/v1`
  API is stable so a direct chart jump normally works, but we have the Phase 0 export as a
  safety net. Verify existing `letsencrypt` ClusterIssuers reconcile and certs stay Ready.
- **ingress-nginx** `infrastructure/base/nginx/release.yaml`: `version: "4.2.5"` → `"4.11.x"`.
  Review `values:` for renamed keys across v1.3→v1.11 (the `controller.service`, `config` block,
  gzip/brotli keys used here are still valid; double-check against the target chart's
  `values.yaml`). Confirm the `LoadBalancer` Service still gets an address via k3s **servicelb**
  (svclb pods) and that existing Ingresses serve TLS.
  - Note: `kubernetes.io/ingress.class` annotation is deprecated in favour of
    `spec.ingressClassName`. Newer ingress-nginx still honours the annotation; migrating Ingresses
    to `ingressClassName: nginx` is optional cleanup, not required for this upgrade.
- Commit each; verify green before proceeding.

## Phase 3 — Step Kubernetes to 1.30 via SUC Plans

Create two Plans under `infrastructure/base/system-upgrade-controller/` (or a sibling dir):
a **server Plan** and an **agent Plan**, following Rancher's k3s-upgrade Plan example:
- `concurrency: 1`, `cordon: true`, `drain: { force: true }`, appropriate
  `nodeSelector`/`match` labels so servers upgrade before agents.
- The k3s version is the Plan's `version:` field (e.g. `v1.28.x+k3s1`). **This is the declarative
  knob** — bumping it in git + commit triggers a rolling upgrade.

Step **one minor at a time** (do not skip minors), from the confirmed current version up to 1.30:

1. `→ v1.28.x+k3s1`: set Plan version, commit, watch SUC roll nodes
   (`kubectl -n system-upgrade get plan,jobs`; `kubectl get nodes` all Ready on new version).
2. **Bump Flux → v2.3.x/2.4.x** once the cluster is on 1.28 (Flux ≥2.3 needs k8s ≥1.28):
   - Upgrade the Flux CLI locally, regenerate `clusters/home/flux-system/gotk-components.yaml`
     (`flux install --export`), commit.
   - Migrate API versions repo-wide:
     - `helm.toolkit.fluxcd.io/v2beta1` → `helm.toolkit.fluxcd.io/v2` in all HelmReleases:
       `infrastructure/base/cert-manager/release.yaml`, `.../nginx/release.yaml`,
       `.../longhorn/release.yaml`, `apps/prod/tesla/release.yaml`,
       `apps/no-deployment/influxdb/release.yaml`.
     - `source.toolkit.fluxcd.io/v1beta2` → `v1` (GitRepository, HelmRepository sources).
     - `kustomize.toolkit.fluxcd.io/v1beta2` → `v1` (`clusters/home/*.yaml`, `gotk-sync.yaml`).
   - Verify all Kustomizations/HelmReleases reconcile after the Flux upgrade before continuing.
3. `→ v1.29.x+k3s1`: bump Plan versions, commit, verify.
4. `→ v1.30.x+k3s1`: bump Plan versions, commit, verify.
5. **Align cert-manager to the 1.30 window**: bump `cert-manager` 1.14 → **1.15.x/1.16.x** and
   switch `installCRDs: true` → `crds.enabled: true` (values key renamed in 1.15). Commit, verify.

## Rollback

- Per-node k3s failure: SUC leaves the node cordoned; k3s keeps prior binaries — restore from the
  Phase 0 etcd snapshot (`k3s server --cluster-reset --cluster-reset-restore-path=...`) on the
  server if the control plane is damaged.
- Component regressions are git-reverts (Flux reconciles back), except cert-manager/ingress-nginx
  CRD schema changes — restore CRs from the Phase 0 export if needed.
- Do **not** attempt to downgrade k8s minors; roll forward or restore the snapshot.
